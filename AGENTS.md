# Blogger 项目 AGENTS 说明

本文件用于约束本项目中的文章编写、分类、渲染与索引更新行为。后续在本仓库中新增或修改文章时，默认遵循本文件。

## 1. 任务目标

在这个项目里，代理最常见的工作是：

- 新增 Markdown 文章
- 调整文章分类
- 优化文章标题层级与目录展示
- 使用项目支持的特殊标记增强内容表现
- 更新文章索引，使新增内容出现在网页中

如果任务和文章内容相关，优先阅读本文件，再执行修改。

## 2. 文章存放规则

所有文章必须放在：

- `Resources/Articles`

当前推荐做法是：

- 所有 Markdown 文章统一直接放在 `Resources/Articles`
- 不再依赖文件夹层级表达模块、章节、专题
- 文章分类、子分类、标签、系列等信息统一写在 Front Matter 中

示例：

- `Resources/Articles/依赖注入.md`
- `Resources/Articles/1.工厂方法模式.md`
- `Resources/Articles/Mahjong流程设计-大厅.md`

注意：

- 索引脚本会递归扫描 `Resources/Articles` 下的 Markdown 文件
- 允许存在历史遗留子目录，但新文章默认不要再按目录分层
- 如果只是为了分类，不要通过新建文件夹表达分类，优先写 Front Matter

## 3. 分类规范

### 3.1 category

`category` 表示长期稳定的大类，例如：

- `C#`
- `C++`
- `Unity`
- `网络编程`
- `设计模式`
- `Program设计文档`

建议：

- 每篇文章都显式写 `category`
- 命名要稳定，避免同义词并存，例如不要同时出现 `前端` 和 `Web前端`

### 3.2 subcategory

`subcategory` 用于表示分类下的专题划分，例如：

- `概念`
- `多线程与异步编程`
- `创建型设计模式`
- `附录`

说明：

- `subcategory` 是可选的
- 页面筛选与统计会直接显示该名称，因此命名要可读、稳定
- 如果文章不需要二级专题，可以不写

### 3.3 文件名

文件名主要用于管理与路径定位，页面展示标题优先取 Front Matter 中的 `title`。

建议：

- 文件名直接表达主题，避免过短或难以区分
- 同级文章需要排序时，可保留数字前缀，例如 `1.`、`2.`、`6.`
- 即使文件名后续调整，只要 `title` 不变，页面展示标题就能保持稳定

### 3.4 Front Matter 推荐字段

推荐每篇文章至少提供以下字段：

```md
---
title: 依赖注入
category: 设计模式
subcategory: 架构设计
tags: [设计模式, 依赖注入, IoC]
series: 架构与解耦
date: 2026-03-12
updatedAt: 2026-03-12
---
```

其中：

- `title`：页面标题
- `category`：一级分类
- `subcategory`：二级分类，可选
- `tags`：标签数组，可选但推荐
- `series`：系列名，可选
- `date` / `updatedAt`：发布日期与更新时间，推荐显式写

## 4. 标题层级规范

网页右侧目录由 Markdown 标题自动提取，当前规则来自 `js/utils.js`：

- 目录一级：`##`
- 目录二级：`###`
- `#` 不进入右侧目录
- `####` 及以下默认不进入右侧目录

因此正文应优先使用：

- `##` 作为主结构标题
- `###` 作为子结构标题

不建议：

- 把正文主结构写成 `#`
- 大量使用 `####` 作为主要层级
- 同一篇文章混用过多编号体系

推荐结构：

```md
## 1. 主题
### 1.1 子主题
```

或：

```md
## 一、主题
### 1. 子主题
```

同一篇文章保持一种风格即可。

## 5. Markdown 渲染规范

Markdown 解析逻辑位于：

- `js/services.js`

默认支持标准 Markdown，并扩展了以下能力。

### 5.1 特殊容器块

支持以下自定义块：

- `:::info`
- `:::warning`
- `:::danger`
- `:::hint`
- `:::abstract`
- `:::ref`

基本写法：

```md
:::info 标题
这里是内容
:::
```

也可以不写标题：

```md
:::warning
这里是内容
:::
```

说明：

- `ref` 在未提供标题时，默认标题为“引用”
- 其他类型不写标题时，标题可为空
- 容器块前后应保留空行，避免解析异常

适用建议：

- `info`：一般说明
- `warning`：注意事项
- `danger`：高风险误区
- `hint`：经验提示
- `abstract`：摘要或结论
- `ref`：引用或参考说明

### 5.2 代码块

项目会对代码块做自定义渲染：

- 显示语言标签
- 显示“复制”按钮
- 自动做 HTML 转义

因此代码块应尽量显式标注语言，例如：

```md
```csharp
public class Demo
{
}
```
```

或：

```md
```js
console.log('hello');
```
```

如果不写语言，展示虽然可用，但高亮结果可能不准确。

### 5.3 标题锚点

标题会自动生成锚点 id，并用于右侧目录跳转。

因此：

- 不要在同一篇文章里大量重复相同标题
- 标题文本尽量清晰稳定

## 6. 索引生成规则

文章列表来自自动生成文件，不是运行时直接扫描目录。

相关文件：

- `scripts/generate-data.js`
- `js/modules_data.js`

规则如下：

- 递归扫描 `Resources/Articles` 下所有 Markdown 文件
- 从文章 Front Matter 中读取 `category`、`subcategory`、`tags`、`series`
- 生成分类、标签、系列、归档等索引数据
- 生成结果写入 `js/modules_data.js`

重要约束：

- `js/modules_data.js` 是自动生成文件，不要手改
- 新增、删除、重命名、移动文章后，必须刷新索引

刷新命令：

```powershell
node .\scripts\generate-data.js
```

或：

```powershell
npm run build
```

## 7. 新增文章的标准流程

新增文章时，按下面顺序执行：

1. 在 `Resources/Articles` 下创建 Markdown 文件
2. 补齐 Front Matter，至少写 `title`、`category`，按需写 `subcategory`、`tags`、`series`
3. 用 `##` / `###` 组织正文标题
4. 需要强调内容时使用 `:::info`、`:::warning` 等特殊块
5. 代码块补充语言标记
6. 运行索引生成命令
7. 确认新文章已进入对应分类、标签或系列列表

## 8. 修改文章时的注意事项

### 8.1 改文件名

会直接影响：

- 同级排序
- 文章路径

如果文章显式写了 `title`，页面展示标题通常不会因改文件名而变化。

改名后必须刷新索引。

### 8.2 改目录位置

会直接影响：

- 文章路径
- 可能影响历史相对路径资源引用

不会再决定文章分类；分类以 Front Matter 为准。

移动后必须刷新索引。

### 8.3 改标题层级

会直接影响文章右侧目录展示。

如果用户明确要求“标题符合目录展示规范”，优先检查是否满足：

- 主结构为 `##`
- 子结构为 `###`

## 9. 当前项目的内容风格建议

结合现有文章，默认风格如下：

- 使用中文写作
- 一篇文章围绕一个主题
- 先讲结论，再讲原因和适用场景
- 结构尽量清晰，方便目录浏览
- 偏概念文章可使用“定义 -> 特点 -> 优缺点 -> 场景 -> 总结”
- 偏实战文章可使用“结论 -> 场景 -> 选型 -> 坑点 -> 建议”

## 10. 维护提醒

### 10.1 不建议的做法

- 手动修改 `js/modules_data.js`
- 通过文件夹层级表达分类
- 用 `#` 或 `####` 作为正文主要层级
- 在同一目录下使用混乱的命名规则
- 缺少 `category` 却希望页面正确分类

### 10.2 设计模式分类提醒

当前 `设计模式` 分类下已经开始承载“架构设计”和“工程实践”类文章。

如果后续此类内容继续增多，建议考虑把附录进一步拆分为更清晰的专题目录，例如：

- `架构设计`
- `工程实践`

以避免 `subcategory` 语义越来越宽。

## 11. 相关文件定位

如果任务涉及文章展示异常，可优先检查：

- `js/services.js`：Markdown 解析、标题提取、特殊容器块
- `js/utils.js`：目录提取层级配置、路径编码
- `scripts/generate-data.js`：文章索引扫描逻辑
- `js/modules_data.js`：自动生成的文章索引结果
- `css/article.css`：文章页样式、目录样式、特殊块样式、代码块样式

如果任务只是新增文章，通常不需要改这些代码文件，优先只改 Markdown 和索引。
