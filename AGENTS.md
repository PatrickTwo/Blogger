# Blogger 项目说明

本文件用于补充该博客项目的内容组织规范、文章编写规范、Markdown 渲染规则与目录生成规则，供后续写作、维护与自动化处理时统一遵循。

## 1. 项目结构总览

项目核心目录如下：

- `Resources/Articles`：所有文章源文件，使用 Markdown 编写
- `js/modules_data.js`：文章索引数据，自动生成，禁止手改
- `scripts/generate-data.js`：扫描文章目录并生成索引
- `js/services.js`：Markdown 解析逻辑，包括目录提取、特殊块解析、代码块渲染
- `js/utils.js`：目录提取配置、路径编码、代码复制等工具
- `css/article.css`：文章页样式，包括标题、目录、代码块、特殊提示块

## 2. 文章存放规则

所有文章必须放在 `Resources/Articles` 下。

当前模块以一级目录区分，例如：

- `Resources/Articles/C#`
- `Resources/Articles/C++`
- `Resources/Articles/Unity`
- `Resources/Articles/设计模式`
- `Resources/Articles/网络编程`
- `Resources/Articles/Program设计文档`
- `Resources/Articles/工具集`
- `Resources/Articles/杂项`

模块下支持两种文章组织方式：

### 2.1 模块根目录直接放文章

例如：

- `Resources/Articles/设计模式/依赖注入.md`

这类文章会被识别为模块下的“直属文章”。

### 2.2 模块下再建章节目录

例如：

- `Resources/Articles/设计模式/B.创建型设计模式/1.工厂方法模式.md`
- `Resources/Articles/设计模式/E.附录/6.MVC、MVP、MVVM架构的异同.md`

这类子目录会被识别为“章节”，章节下的 `.md` 文件会显示为该章节中的文章。

注意：

- 当前索引生成脚本只扫描两层结构
- 也就是说支持“模块 -> 文章”与“模块 -> 章节 -> 文章”
- 不要继续向下再套更深层目录作为正文文章目录，否则前端不会自动收录更深层级内容

## 3. 文章分类规范

新增文章时，应优先放入已有模块与已有章节中，保持分类稳定。

### 3.1 模块层级

模块代表一个较大的知识领域，例如：

- `设计模式`
- `Unity`
- `C#`
- `网络编程`

除非确认是新的长期知识域，否则不要随意新增模块。

### 3.2 章节层级

章节适合表示某一模块下的稳定专题，例如：

- `A.设计原则与设计模式概述`
- `B.创建型设计模式`
- `C.结构型设计模式`
- `D.行为型模式`
- `E.附录`

建议保持统一的排序前缀，例如：

- `A.`
- `B.`
- `C.`
- `1.`
- `2.`

这样可以让文件系统中的排序与页面展示顺序保持一致。

### 3.3 文件命名规范

文章标题直接来源于文件名去掉 `.md` 后的结果，因此文件名就是页面展示标题。

建议使用以下命名方式：

- `概述.md`
- `依赖注入.md`
- `1.工厂方法模式.md`
- `6.MVC、MVP、MVVM架构的异同.md`

建议：

- 需要排序的同级文章使用数字前缀
- 数字前缀后使用 `.`，与当前项目已有内容保持一致
- 文件名应尽量直接表达文章主题，避免过短或含糊命名

## 4. 索引生成规则

文章列表不是运行时动态遍历目录，而是预先生成到 `js/modules_data.js` 中。

生成逻辑位于：

- `scripts/generate-data.js`

规则如下：

- 扫描 `Resources/Articles` 下所有一级目录作为模块
- 扫描模块根目录下的 `.md` 文件作为直属文章
- 扫描模块下一层子目录，若子目录中存在 `.md` 文件，则视为章节
- 章节目录名称会直接显示在前端列表中
- 文件路径会被写入 `js/modules_data.js`

新增、删除、移动文章后，必须重新生成索引。

可用命令：

```powershell
node .\scripts\generate-data.js
```

或：

```powershell
npm run build
```

注意：

- `js/modules_data.js` 为自动生成文件，不应手动编辑
- 如果改了文章但没刷新索引，前端列表中可能不会出现新文章

## 5. 网页解析规范

Markdown 解析逻辑位于 `js/services.js`，目前项目支持标准 Markdown，并有若干自定义行为。

### 5.1 标题目录提取规则

文章右侧目录由标题自动生成，当前配置位于 `js/utils.js`：

- 一级目录标题提取自 `##`
- 二级目录标题提取自 `###`
- 默认展示二级目录

也就是说，当前目录提取配置等价于：

- `##` 作为目录一级项
- `###` 作为目录二级项
- `#` 不进入右侧目录
- `####` 及以下默认不进入右侧目录

因此写文章时建议：

- 正文主结构统一使用 `##`
- 子结构统一使用 `###`
- 非必要不要混用过多 `####`

这样目录展示最稳定，也最符合当前站点风格。

### 5.2 标题锚点规则

标题会被自动转为页面锚点 `id`，规则大致为：

- 转小写
- 去除部分 Markdown 强调符号
- 将非字母、数字、中文的连续字符替换为 `-`

因此：

- 标题文本最好清晰、稳定
- 不要在同一篇文章中重复大量完全相同的标题

否则可能导致锚点重复，影响目录跳转。

### 5.3 段落与换行

当前 `marked.parse` 启用了 `breaks: true`，因此普通换行会更接近 Markdown 中的软换行转显示换行。

写作时仍建议：

- 段落之间保留空行
- 列表、引用、代码块与正文之间保留空行

避免不同 Markdown 结构粘连导致解析不稳定。

## 6. 可用的特殊标记

项目支持自定义容器块语法，定义在 `js/services.js` 中。

支持的特殊标记如下：

- `:::info`
- `:::warning`
- `:::danger`
- `:::hint`
- `:::abstract`
- `:::ref`

语法格式：

```md
:::info 标题
这里是内容
:::
```

也支持不写标题：

```md
:::warning
这里是内容
:::
```

### 6.1 各标记说明

#### `:::info`

用于一般说明、补充信息。

#### `:::warning`

用于警告、注意事项、易错点。

#### `:::danger`

用于高风险提醒、严重误区、不建议做法。

#### `:::hint`

用于经验提示、技巧、实践建议。

#### `:::abstract`

用于摘要、前情提要、核心结论块。

#### `:::ref`

用于引用、参考说明。

当 `ref` 未提供标题时，系统默认标题为“引用”。

### 6.2 特殊标记使用建议

- 容器块前后都保留空行
- 容器内容尽量为完整段落、列表或小段说明
- 不要在一个段落中间嵌入容器块
- 容器块适合强调信息，不要滥用，否则会破坏文章阅读节奏

## 7. 代码块规范

项目对代码块做了自定义渲染：

- 会显示语言标签
- 会显示“复制”按钮
- 会自动做 HTML 转义，避免代码中的 `<tag>` 被当作 HTML

建议所有代码块都显式标注语言，例如：

```md
```csharp
public class Demo
{
}
```
```

或：

```md
```js
console.log('hello');
```
```

如果未标注语言，显示层会按 `text` 处理；高亮阶段在没有类名时会默认补成 `language-csharp`，因此不写语言可能导致前端高亮风格不准确。

## 8. 表格、列表、引用与图片

项目当前对以下标准 Markdown 元素有样式支持：

- 无序列表
- 有序列表
- 引用块
- 表格
- 行内代码
- 代码块
- 图片
- 链接

建议：

- 表格用于对比信息
- 列表用于并列要点
- 引用用于说明外部观点或总结性语句
- 图片路径需要能被站点访问到，否则无法正常展示

## 9. 文章写作风格建议

结合当前站点现有文章，建议保持以下风格一致性：

- 使用中文为主
- 标题层级清晰
- 一篇文章围绕一个明确主题展开
- 多使用“定义 -> 问题 -> 方案 -> 优缺点 -> 场景 -> 总结”的结构
- 实战型文章优先给结论，再展开解释

对于目录展示友好性，建议采用：

- `## 一、xxx`
- `### 1. xxx`

或者：

- `## 1. xxx`
- `### 1.1 xxx`

同一篇文章内最好保持一种编号体系，不要混用过多风格。

## 10. 当前前端展示行为补充

### 10.1 页面标题来源

文章页顶部标题来自路由中的 `title`，本质上来源于文件名。

因此：

- 文件名就是展示名
- 如果想改文章显示标题，通常需要改文件名并刷新索引

### 10.2 目录显示范围

文章右侧目录只展示被提取到的标题。

当前默认只覆盖：

- `##`
- `###`

所以如果一篇文章全用 `####` 开始，目录很可能不完整甚至不显示。

### 10.3 本地文件路径处理

文章路径在前端会按 `/` 分段并做 URL 编码，因此中文路径、空格路径都能正常处理。

但仍建议：

- 不要使用过于怪异的特殊字符
- 文件与目录命名尽量保持可读性

## 11. 内容维护注意事项

### 11.1 新增文章后要做的事

1. 将 Markdown 放到正确分类目录
2. 检查文件名是否符合展示与排序规则
3. 检查正文标题是否以 `##` / `###` 为主
4. 运行索引生成命令
5. 确认文章已出现在对应模块或章节列表

### 11.2 调整分类时要注意

- 移动文章路径后必须刷新索引
- 修改章节目录名会影响前端列表显示
- 修改文件名会直接影响文章标题显示与排序

### 11.3 不建议的做法

- 手动编辑 `js/modules_data.js`
- 在文章目录中使用过深层级目录
- 大量使用 `####` 作为正文主层级
- 让同一模块下命名风格严重不统一

## 12. 架构与内容组织提醒

当前 `设计模式` 模块下已经包含一部分偏“架构设计”和“工程实践”的内容，而不仅仅是 GoF 设计模式。

如果未来此类文章继续增多，建议考虑把附录进一步拆分为更明确的专题目录，例如：

- `E.架构设计`
- `F.工程实践`

这样能避免“附录”目录语义过宽，后续更利于维护与检索。

## 13. 给后续协作者的执行建议

如果任务是“新增一篇文章”，优先按以下顺序执行：

1. 找到最合适的模块与章节目录
2. 按现有排序规则命名文件
3. 正文使用 `##` 和 `###` 组织结构
4. 需要强调内容时优先使用 `:::info`、`:::warning` 等特殊块
5. 代码块显式写语言标记
6. 执行索引生成命令
7. 最后再检查文章是否已进入列表

如果任务是“修改文章展示规则”，优先检查：

- `js/services.js`
- `js/utils.js`
- `css/article.css`
- `scripts/generate-data.js`

以上文件共同决定了文章渲染、目录提取、样式表现与索引生成行为。
