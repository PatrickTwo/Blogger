# Blogger 项目 AGENTS 说明

本文件用于约束本项目中的文章编写、分类、渲染与索引更新行为。后续在本仓库中新增或修改文章时，默认遵循本文件。

## 1. 任务目标

在这个项目里，代理最常见的工作是：

- 新增 Markdown 文章
- 调整文章分类
- 优化文章标题层级与目录展示
- 使用项目支持的特殊标记增强内容表现
- 更新文章索引，使新增内容出现在网页中

如果任务和文章内容相关，优先阅读本文件，再执行修改。

## 2. 文章存放规则

所有文章必须放在：

- `Resources/Articles`

当前文章目录只支持两种结构：

### 2.1 模块直属文章

示例：

- `Resources/Articles/设计模式/依赖注入.md`

### 2.2 模块 -> 章节 -> 文章

示例：

- `Resources/Articles/设计模式/B.创建型设计模式/1.工厂方法模式.md`
- `Resources/Articles/设计模式/E.附录/6.MVC、MVP、MVVM架构的异同.md`

注意：

- 当前索引脚本只扫描到“模块下一层子目录”
- 不要把正文文章放到更深层目录中，否则前端不会自动收录

## 3. 分类规范

### 3.1 模块

模块代表长期稳定的大类，例如：

- `C#`
- `C++`
- `Unity`
- `网络编程`
- `设计模式`
- `Program设计文档`

除非主题已经形成新的长期栏目，否则优先放入已有模块。

### 3.2 章节

章节用于模块下的专题划分，例如：

- `A.设计原则与设计模式概述`
- `B.创建型设计模式`
- `C.结构型设计模式`
- `D.行为型模式`
- `E.附录`

章节名会直接显示在网页列表中，因此命名要稳定、可读、可排序。

### 3.3 文件名

文件名就是文章展示标题来源，页面会直接去掉 `.md` 后显示。

建议：

- 同级文章需要排序时，加数字前缀，例如 `1.`、`2.`、`6.`
- 文件名直接表达主题，不要过短或过于口语化
- 新文章命名风格要尽量和当前目录已有文章保持一致

## 4. 标题层级规范

网页右侧目录由 Markdown 标题自动提取，当前规则来自 `js/utils.js`：

- 目录一级：`##`
- 目录二级：`###`
- `#` 不进入右侧目录
- `####` 及以下默认不进入右侧目录

因此正文应优先使用：

- `##` 作为主结构标题
- `###` 作为子结构标题

不建议：

- 把正文主结构写成 `#`
- 大量使用 `####` 作为主要层级
- 同一篇文章混用过多编号体系

推荐结构：

```md
## 1. 主题
### 1.1 子主题
```

或：

```md
## 一、主题
### 1. 子主题
```

同一篇文章保持一种风格即可。

## 5. Markdown 渲染规范

Markdown 解析逻辑位于：

- `js/services.js`

默认支持标准 Markdown，并扩展了以下能力。

### 5.1 特殊容器块

支持以下自定义块：

- `:::info`
- `:::warning`
- `:::danger`
- `:::hint`
- `:::abstract`
- `:::ref`

基本写法：

```md
:::info 标题
这里是内容
:::
```

也可以不写标题：

```md
:::warning
这里是内容
:::
```

说明：

- `ref` 在未提供标题时，默认标题为“引用”
- 其他类型不写标题时，标题可为空
- 容器块前后应保留空行，避免解析异常

适用建议：

- `info`：一般说明
- `warning`：注意事项
- `danger`：高风险误区
- `hint`：经验提示
- `abstract`：摘要或结论
- `ref`：引用或参考说明

### 5.2 代码块

项目会对代码块做自定义渲染：

- 显示语言标签
- 显示“复制”按钮
- 自动做 HTML 转义

因此代码块应尽量显式标注语言，例如：

```md
```csharp
public class Demo
{
}
```
```

或：

```md
```js
console.log('hello');
```
```

如果不写语言，展示虽然可用，但高亮结果可能不准确。

### 5.3 标题锚点

标题会自动生成锚点 id，并用于右侧目录跳转。

因此：

- 不要在同一篇文章里大量重复相同标题
- 标题文本尽量清晰稳定

## 6. 索引生成规则

文章列表来自自动生成文件，不是运行时直接扫描目录。

相关文件：

- `scripts/generate-data.js`
- `js/modules_data.js`

规则如下：

- 扫描 `Resources/Articles` 下一级目录作为模块
- 扫描模块根目录中的 `.md` 文件作为直属文章
- 扫描模块下一层子目录中的 `.md` 文件作为章节文章
- 生成结果写入 `js/modules_data.js`

重要约束：

- `js/modules_data.js` 是自动生成文件，不要手改
- 新增、删除、重命名、移动文章后，必须刷新索引

刷新命令：

```powershell
node .\scripts\generate-data.js
```

或：

```powershell
npm run build
```

## 7. 新增文章的标准流程

新增文章时，按下面顺序执行：

1. 判断应该放入哪个模块、哪个章节
2. 按当前目录风格命名文件
3. 用 `##` / `###` 组织正文标题
4. 需要强调内容时使用 `:::info`、`:::warning` 等特殊块
5. 代码块补充语言标记
6. 运行索引生成命令
7. 确认新文章已进入对应列表

## 8. 修改文章时的注意事项

### 8.1 改文件名

会直接影响：

- 页面显示标题
- 同级排序
- 文章路径

改名后必须刷新索引。

### 8.2 改目录位置

会直接影响：

- 所属模块
- 所属章节
- 前端列表展示位置

移动后必须刷新索引。

### 8.3 改标题层级

会直接影响文章右侧目录展示。

如果用户明确要求“标题符合目录展示规范”，优先检查是否满足：

- 主结构为 `##`
- 子结构为 `###`

## 9. 当前项目的内容风格建议

结合现有文章，默认风格如下：

- 使用中文写作
- 一篇文章围绕一个主题
- 先讲结论，再讲原因和适用场景
- 结构尽量清晰，方便目录浏览
- 偏概念文章可使用“定义 -> 特点 -> 优缺点 -> 场景 -> 总结”
- 偏实战文章可使用“结论 -> 场景 -> 选型 -> 坑点 -> 建议”

## 10. 维护提醒

### 10.1 不建议的做法

- 手动修改 `js/modules_data.js`
- 把文章放到超过两层的深目录
- 用 `#` 或 `####` 作为正文主要层级
- 在同一目录下使用混乱的命名规则

### 10.2 设计模式模块提醒

当前 `设计模式/E.附录` 已经开始承载“架构设计”和“工程实践”类文章。

如果后续此类内容继续增多，建议考虑把附录进一步拆分为更清晰的专题目录，例如：

- `E.架构设计`
- `F.工程实践`

以避免“附录”语义越来越宽。

## 11. 相关文件定位

如果任务涉及文章展示异常，可优先检查：

- `js/services.js`：Markdown 解析、标题提取、特殊容器块
- `js/utils.js`：目录提取层级配置、路径编码
- `scripts/generate-data.js`：文章索引扫描逻辑
- `js/modules_data.js`：自动生成的文章索引结果
- `css/article.css`：文章页样式、目录样式、特殊块样式、代码块样式

如果任务只是新增文章，通常不需要改这些代码文件，优先只改 Markdown 和索引。
