# 项目讲解文档

## 1. 文档目标

本文档用于帮助你从“项目整体结构”和“代码运行流程”两个角度理解当前博客项目。

这份文档重点回答几个问题：

- 这个项目是怎么组织的
- 一篇 Markdown 文章是如何变成网页内容的
- 构建期和运行时分别做了什么
- 前端各个核心文件分别负责什么
- 这次重构后的代码应该按什么顺序阅读

如果你的目标是学习这套代码，建议不要一上来就逐个文件硬读，而是先看本文档，再按文中的推荐顺序去看源码。

## 2. 项目定位

当前项目本质上是一个：

- 基于 `Vue 3 + Vue Router`
- 使用 `Hash 路由`
- 以 `Markdown` 作为内容源
- 通过本地构建脚本预生成站点数据
- 再由前端静态页面负责展示

的个人博客系统。

它不是传统的“后端动态博客”，也不是完整的 SSR 框架项目，而是“静态资源 + 构建脚本 + 前端单页应用”的组合。

你可以把它拆成两部分来理解：

1. 构建期
把 `Resources/Articles` 和 `Resources/Pages` 里的内容扫描、解析、整理，最终生成前端可以直接使用的数据文件和站点输出文件。

2. 运行时
浏览器加载 `index.html` 后，由 Vue 应用接管页面，根据路由读取预生成的数据，再去拉取对应的 Markdown 正文并渲染出来。

## 4. 整体运行流程

理解这个项目最重要的一条主线是：

`Markdown 内容 -> 构建脚本解析 -> 生成站点数据 -> Vue 路由读取数据 -> 页面再拉取正文并渲染`

可以展开成下面这 7 步：

1. 你在 `Resources/Articles` 中编写 Markdown 文章。
2. 可选地在文章顶部写 Front Matter 元数据。
3. 执行构建命令，例如 `npm run build`。
4. 构建脚本扫描文章和独立页面，整理出模块、文章、标签、系列、归档等结构化数据。
5. 构建脚本输出 [modules_data.js](C:\Users\Hetu\OneDrive\Blogger\js\modules_data.js)、[rss.xml](C:\Users\Hetu\OneDrive\Blogger\rss.xml)、[sitemap.xml](C:\Users\Hetu\OneDrive\Blogger\sitemap.xml)、[robots.txt](C:\Users\Hetu\OneDrive\Blogger\robots.txt)。
6. 浏览器打开 [index.html](C:\Users\Hetu\OneDrive\Blogger\index.html) 后，Vue 应用启动并读取 `modules_data.js`。
7. 当用户进入文章页时，前端再根据文章路径去 fetch 对应的 Markdown 文件，并在浏览器里转成 HTML。

这里有一个很重要的设计点：

- “文章列表、标签、系列、归档”等聚合信息是在构建期完成的
- “文章正文 Markdown 转 HTML”是在运行时完成的

这意味着当前项目采用的是“半静态”思路，而不是把所有文章正文在构建期直接输出成 HTML 页面。

## 5. 构建期代码讲解

如果你想理解“数据从哪里来”，优先看构建脚本。

### 5.1 入口脚本

[build-site.js](C:\Users\Hetu\OneDrive\Blogger\scripts\build-site.js) 是统一构建入口。

它做的事情很简单：

- 调用 `buildSite()`
- 打印构建结果
- 如果有警告就输出警告摘要
- 如果出错就退出进程

你可以把它理解成“命令行启动器”，真正的核心逻辑不在这里，而在 [site-builder.js](C:\Users\Hetu\OneDrive\Blogger\scripts\site-builder.js)。

### 5.2 构建核心

[site-builder.js](C:\Users\Hetu\OneDrive\Blogger\scripts\site-builder.js) 是整个项目最核心的文件之一。

这个文件主要负责 4 类事情：

1. 读取内容
扫描 `Articles`、`Pages`、`site.config.json`

2. 解析内容
解析 Front Matter、日期、标签、系列、标题、摘要、阅读时长、锚点等信息

3. 生成聚合数据
生成模块、文章、标签、系列、归档、独立页面等前端数据结构

4. 输出结果
写入 `modules_data.js`、`rss.xml`、`sitemap.xml`、`robots.txt`

### 5.3 Front Matter 解析

重构后的一个关键点是加入了 Front Matter 能力。

例如文章顶部可以写：

```md
---
title: MVC、MVP、MVVM架构的异同
summary: 对三种经典界面架构做概念和实践层面的对比。
tags: [架构设计, 设计模式, MVC, MVP, MVVM]
series: 界面架构
order: 6
date: 2026-03-26
updatedAt: 2026-03-26
draft: false
---
```

`site-builder.js` 和 [utils.js](C:\Users\Hetu\OneDrive\Blogger\js\utils.js) 里都实现了 Front Matter 解析逻辑。

这里分成两套是有原因的：

- 构建期需要解析 Front Matter 来生成站点索引
- 运行时也需要解析 Front Matter，因为前端直接 fetch 到的是原始 Markdown，渲染正文时必须把头部元数据剥掉

这是一个典型的“同一规则在构建期和运行时都要用”的场景。

### 5.4 为什么要生成 `modules_data.js`

[modules_data.js](C:\Users\Hetu\OneDrive\Blogger\js\modules_data.js) 可以理解为“前端的静态数据库”。

它把原本分散在目录结构和 Markdown 文件里的信息，整理成了前端能直接消费的 JS 常量，例如：

- 站点元信息
- 模块列表
- 文章列表
- 标签列表
- 系列列表
- 归档数据
- 独立页面列表

这样做的好处是：

- 前端首页、列表页、标签页、系列页不用自己扫描目录
- 页面首次渲染时不需要做复杂的数据整理
- 搜索、归档、筛选都能直接在内存中完成

从架构角度看，这相当于用构建脚本把“内容源”预处理成了一个“前端查询友好型数据层”。

### 5.5 内容校验脚本

[check-content.js](C:\Users\Hetu\OneDrive\Blogger\scripts\check-content.js) 用于执行内容校验。

它本身也很薄，只是调用 `checkContent()`，真正的校验逻辑仍然放在 `site-builder.js` 中。

校验内容主要包括：

- 重复标题
- 缺失元数据的降级情况
- 无效相对路径
- 无效锚点
- 系列顺序问题

这类设计是合理的，因为：

- `buildSite()` 负责产出站点
- `checkContent()` 负责独立校验
- 但它们共享相同的底层解析逻辑

这样能避免规则写两遍。

## 6. 运行时代码讲解

如果你想理解“页面为什么会这样显示”，重点看前端运行时代码。

### 6.1 应用入口

[index.html](C:\Users\Hetu\OneDrive\Blogger\index.html) 负责三件事：

- 加载 Vue、Vue Router、marked、highlight.js 等依赖
- 提供整体页面骨架，例如头部、主内容区、底部
- 挂载 Vue 应用

这里有一个你最近刚修过的重点：

- `router-view` 外层包了一层真实 DOM 容器
- 并使用 `route.fullPath` 作为切换 key

这是为了解决“带 fragment 根节点的页面组件放进 `<Transition>` 后，切页正文空白”的问题。

### 6.2 路由与应用状态

[app.js](C:\Users\Hetu\OneDrive\Blogger\js\app.js) 负责：

- 注册路由
- 管理顶层应用状态
- 根据当前路由构建 SEO 信息
- 在路由切换后触发统计和元信息更新

这里最值得学习的是两个点：

1. 路由只负责切页面，不负责复杂的数据处理
2. 页面级 SEO 和统计统一挂在 `router.afterEach()`

这让职责比较清晰：

- 页面组件专注展示
- 服务层负责 SEO、统计
- 路由层负责切换后的统一副作用

### 6.3 组件层

[components.js](C:\Users\Hetu\OneDrive\Blogger\js\components.js) 是当前前端的“主战场”。

它包含了几乎所有页面和通用组件，例如：

- `Breadcrumb`
- `SearchBox`
- `HomeView`
- `ListView`
- `SearchResultsView`
- `ArticleView`
- `ArchiveView`
- `TagsView`
- `SeriesView`
- `AboutView`
- `NotFoundView`

你可以把这个文件理解为：

- 上半部分是工具函数和通用逻辑
- 中间是通用组件
- 下半部分是各个页面组件

建议阅读时重点看这几条线：

1. `findArticlesByKeyword()`
看搜索是如何同时匹配标题、摘要、标签、模块和系列的。

2. `useMarkdownContent()`
看页面是如何根据 Markdown 路径去加载正文、处理错误、提取目录、渲染代码高亮的。

3. `ArticleView`
看文章页如何把正文、目录、上一篇下一篇、系列文章、相关推荐拼起来。

4. `ListView` / `TagsView` / `SeriesView`
看这些聚合页如何消费 `modules_data.js` 里的预生成数据。

### 6.4 服务层

[services.js](C:\Users\Hetu\OneDrive\Blogger\js\services.js) 提供了三类核心服务：

- `MarkdownService`
- `SeoService`
- `AnalyticsService`

#### 6.4.1 MarkdownService

这是运行时最关键的服务。

它负责：

- 解析 Front Matter
- 解析 Markdown 正文
- 提取目录
- 自定义标题锚点
- 自定义代码块渲染
- 处理特殊容器块
- 相对资源路径修正
- 代码高亮

这里要特别注意两点：

第一，目录不是靠页面组件自己扫 DOM 得到的，而是在 Markdown 解析阶段顺手提取出来的。  
第二，项目支持了很多自定义块，例如：

- `:::info`
- `:::warning`
- `:::danger`
- `:::hint`
- `:::abstract`
- `:::ref`
- `:::summary`
- `:::updates`

这说明当前项目已经不只是“标准 Markdown 渲染器”，而是一个带内容增强规则的博客渲染系统。

#### 6.4.2 SeoService

它的职责很单一：

- 根据当前页面更新浏览器标题
- 更新 `description`
- 更新 `keywords`
- 更新 `og:*` 元信息

这是一个很典型的“页面副作用服务”。

#### 6.4.3 AnalyticsService

它负责统计页面访问。

当前支持两种思路：

- 本地统计
- 预留 GoatCounter 接入能力

本地统计的特点是简单、无外部依赖，适合个人博客的轻量使用场景。

### 6.5 工具层

[utils.js](C:\Users\Hetu\OneDrive\Blogger\js\utils.js) 负责一些跨页面复用的通用逻辑，例如：

- 标题层级配置
- 路径编码
- slug 生成
- Markdown 纯文本提取
- Front Matter 解析
- 锚点链接拼接
- 主题切换
- 代码复制

这里最值得注意的是 `TOC_CONFIG`：

- 右侧目录一级标题来自 `##`
- 右侧目录二级标题来自 `###`

这就是为什么你之前写文章时，需要专门控制标题层级来适配目录显示规范。

## 9. 几个关键设计点

这次重构里，有几个设计点特别值得你关注。

### 9.1 构建期和运行时分工清晰

不是所有事情都放到浏览器里做，而是把“聚合型工作”前置到构建脚本。

这能明显减轻前端运行时复杂度，是这次重构最重要的改进之一。

### 9.2 内容模型正式化

Front Matter 的引入，让文章不再完全依赖文件名驱动。

这一步一旦完成，后续很多能力都更容易做：

- 标签页
- 系列页
- 归档页
- RSS
- Sitemap
- SEO
- 推荐文章

### 9.3 Markdown 渲染系统已经有“博客规则”

这个项目并不是把 `marked` 直接拿来就用，而是在渲染前后加了很多自定义逻辑。

这意味着你以后如果要扩展内容能力，优先应该去 [services.js](C:\Users\Hetu\OneDrive\Blogger\js\services.js) 里扩，而不是到页面组件里做各种临时处理。

### 9.4 当前架构仍有一个可继续优化的点

现在 [components.js](C:\Users\Hetu\OneDrive\Blogger\js\components.js) 文件已经很大，页面组件、通用组件、辅助函数都堆在一个文件里。

这对中小项目还能接受，但继续扩展时会越来越难维护。后续如果继续迭代，我建议把它拆成：

- `components/common`
- `components/views`
- `composables`

也就是把“页面组件”和“可复用逻辑”分开。

## 10. 学习这份代码时的建议

最后给你几个更实用的学习建议。

### 10.1 不要把 `modules_data.js` 当作手写源码学习

这个文件主要是构建产物，更适合拿来观察“最终数据长什么样”，不适合逐行精读。

### 10.2 真正的核心要看这 4 个地方

- [scripts/site-builder.js](C:\Users\Hetu\OneDrive\Blogger\scripts\site-builder.js)
- [js/app.js](C:\Users\Hetu\OneDrive\Blogger\js\app.js)
- [js/services.js](C:\Users\Hetu\OneDrive\Blogger\js\services.js)
- [js/components.js](C:\Users\Hetu\OneDrive\Blogger\js\components.js)

### 10.3 遇到“为什么这里这样做”，优先从职责边界思考

例如：

- 为什么 Front Matter 解析写了两次
- 为什么标签和系列要在构建期生成
- 为什么 SEO 更新放在路由后置钩子
- 为什么正文运行时再 fetch Markdown

这些问题的答案，本质上都和“构建期职责”和“运行时职责”的边界有关。

### 10.4 建议边读边做小实验

比如你可以自己试这些改动：

1. 给一篇旧文章补 Front Matter，再重新构建，看 `modules_data.js` 发生了什么变化。
2. 新增一个 `:::summary` 或 `:::updates` 块，看文章页渲染效果。
3. 给某篇文章增加 `series`，再看系列页和文章相关推荐如何变化。
4. 修改 `TOC_CONFIG`，观察右侧目录层级变化。

这种方式会比单纯看代码快很多。
