在界面开发里，MVC、MVP、MVVM 经常被放在一起比较。它们都在解决同一个核心问题：如何把界面展示、业务逻辑、数据状态拆开，降低耦合，提高可维护性。

## 1. 为什么会出现这三种架构

早期项目里，最常见的问题是把界面更新、输入处理、业务判断、数据存储全部写在同一个类里。这样写一开始很快，但随着需求增加，会逐渐出现下面几个问题：

- UI 脚本越来越臃肿
- 逻辑难以复用
- 测试成本高
- 修改一个小需求，容易牵动整片代码

MVC、MVP、MVVM 本质上都是为了解决“界面层代码失控”而产生的分层方案。

## 2. 三者的核心角色分别是什么

### 2.1 MVC

- Model：负责数据和业务规则
- View：负责界面展示
- Controller：负责接收输入、协调 Model 与 View

MVC 的典型思路是：用户操作 View，Controller 接收操作后修改 Model，再决定如何刷新 View。

### 2.2 MVP

- Model：负责数据和业务规则
- View：负责界面展示，通常只暴露接口能力
- Presenter：负责界面逻辑与业务编排

MVP 的关键点在于 Presenter 会直接驱动 View。也就是说，View 尽量“被动”，不自己处理复杂逻辑。

### 2.3 MVVM

- Model：负责数据和业务规则
- View：负责界面展示
- ViewModel：负责把界面需要的数据和命令组织出来

MVVM 的核心思想是数据绑定。View 不再频繁手写“从逻辑层取值再赋给控件”的代码，而是通过绑定机制，让 View 随着 ViewModel 的状态变化自动刷新。

## 3. 三者的共同点

- 都强调职责拆分，避免一个类包办所有事情
- 都希望让界面层与业务层解耦
- 都把数据与展示分离，提升维护性
- 都适合中大型界面系统，不适合继续用“一个脚本写到底”的方式

所以不要把它们理解成互相完全对立的架构，它们更像是同一类问题下的三种演进方案。

## 4. 三者最核心的区别

### 4.1 控制界面逻辑的角色不同

- MVC 由 Controller 承担协调职责
- MVP 由 Presenter 承担界面逻辑
- MVVM 由 ViewModel 承担状态组织与命令封装

### 4.2 View 是否足够被动不同

- MVC 中，View 往往不是完全被动，实际项目里经常会带一点逻辑
- MVP 中，View 通常最被动，只负责显示和转发事件
- MVVM 中，View 负责绑定和展示，逻辑更多转移到 ViewModel

### 4.3 对数据绑定机制的依赖程度不同

- MVC 通常不依赖数据绑定
- MVP 一般也不强依赖数据绑定，更多依赖 Presenter 手动更新 View
- MVVM 高度依赖数据绑定、通知机制、命令机制

### 4.4 可测试性的重点不同

- MVC 中 Controller 可以测试，但 View 与 Controller 边界有时不够清晰
- MVP 中 Presenter 最适合做单元测试，因为 View 往往是接口
- MVVM 中 ViewModel 最适合测试，因为它通常不依赖具体 UI 控件

## 5. 用一句话理解三者

- MVC：Controller 管事
- MVP：Presenter 盯着 View 干活
- MVVM：View 绑定 ViewModel 自动更新

如果只想快速抓住差异，记住这三句话通常就够用了。

## 6. 一个登录界面的三种写法思路

假设我们要实现一个简单登录界面，包含用户名输入框、密码输入框、登录按钮、提示文本。

### 6.1 MVC 的思路

- View 负责展示输入框和按钮
- Controller 读取用户输入
- Controller 调用 Model 校验账号密码
- Controller 决定 View 显示成功还是失败

特点是流程清晰，但 Controller 容易变胖。

### 6.2 MVP 的思路

- View 只提供 `GetUserName`、`GetPassword`、`SetMessage` 这类接口
- Presenter 监听登录事件
- Presenter 调用 Model 处理登录
- Presenter 再调用 View 接口更新界面

特点是 View 很薄，Presenter 很清晰，适合手动控制较强的 UI 项目。

### 6.3 MVVM 的思路

- View 绑定 `UserName`、`Password`、`Message`
- 登录按钮绑定 `LoginCommand`
- ViewModel 执行登录逻辑并更新 `Message`
- View 因为绑定机制自动刷新

特点是声明式更强，界面刷新代码更少，但前提是框架对绑定支持足够好。

## 7. 三者各自的优缺点

### 7.1 MVC 的优缺点

优点：

- 概念直观，容易入门
- 适合中小型项目快速分层
- 很多 Web 框架都能看到它的影子

缺点：

- Controller 容易承担过多职责
- 复杂界面下，View 和 Controller 的边界容易模糊
- 在高交互 UI 中，后期维护成本可能上升

### 7.2 MVP 的优缺点

优点：

- View 与逻辑分离更明显
- Presenter 可测试性强
- 适合 UI 逻辑复杂、但绑定能力一般的项目

缺点：

- View 接口数量容易膨胀
- Presenter 容易变成“大管家”
- 页面很多时，接口和同步代码较繁琐

### 7.3 MVVM 的优缺点

优点：

- 数据绑定后，界面更新代码大幅减少
- ViewModel 更利于测试与复用
- 很适合状态驱动型界面

缺点：

- 对框架能力要求较高
- 绑定链路过多时，调试成本会增加
- 初学者容易把业务逻辑、UI 状态逻辑、领域逻辑都堆进 ViewModel

## 8. 适用场景怎么选

### 8.1 适合 MVC 的场景

- 项目规模不大
- 页面交互复杂度一般
- 团队更熟悉传统 Controller 驱动模式

### 8.2 适合 MVP 的场景

- 需要较强的可测试性
- UI 逻辑比较复杂
- 框架本身不擅长数据绑定

例如一些传统客户端、早期 Android 页面式开发、部分游戏工具界面。

### 8.3 适合 MVVM 的场景

- 框架有成熟的数据绑定机制
- 页面状态多、联动多
- 想减少手写界面同步代码

例如 WPF、Vue、部分 Unity UI Toolkit 思路、现代前端状态驱动开发。

## 9. 三者之间的关系更像演进，不是替代

从思想上看，可以把它们理解为一条逐步演进的路线：

- MVC 先把职责拆开
- MVP 进一步让 View 变得更被动
- MVVM 再进一步借助绑定机制，把“更新 View”这件事自动化

所以它们不是谁绝对先进、谁绝对落后，而是适合的技术环境不同。

## 10. 实际开发中最容易踩的坑

### 10.1 把架构名词当成结果

很多项目嘴上说用了 MVVM，实际上只是把原来臃肿的 UI 脚本改名成了 ViewModel，职责并没有真的分开。

### 10.2 ViewModel 或 Presenter 过胖

如果把网络请求、缓存、领域规则、界面状态、动画控制全部塞进 Presenter 或 ViewModel，最终只是把“胖 View”换成了“胖中间层”。

### 10.3 分层方向不干净

例如：

- View 直接操作数据存储
- Model 直接引用具体 UI 控件
- Presenter 或 ViewModel 直接写死场景对象路径

这些问题会破坏分层，让架构名存实亡。

### 10.4 为了架构而架构

小项目、一次性工具、简单页面，如果交互并不复杂，过度引入抽象层只会增加理解成本。

## 11. 一张对比表快速总结

| 维度 | MVC | MVP | MVVM |
| --- | --- | --- | --- |
| 核心协调者 | Controller | Presenter | ViewModel |
| View 是否被动 | 一般 | 强被动 | 偏被动 |
| UI 更新方式 | 手动协调 | Presenter 手动更新 | 数据绑定自动更新 |
| 测试友好度 | 中 | 高 | 高 |
| 学习成本 | 低 | 中 | 中到高 |
| 框架依赖 | 低 | 低到中 | 高 |
| 适合界面复杂度 | 中 | 中到高 | 高 |

## 12. 结论

如果团队还在解决“界面逻辑不要和业务逻辑写在一起”的问题，MVC 是一个容易上手的起点。

如果项目更强调测试性、界面逻辑控制、明确的职责分离，MVP 会更稳。

如果所用技术栈具备成熟绑定机制，并且页面状态联动很多，MVVM 往往能带来更高的开发效率。

从架构设计角度看，真正重要的不是你给它起名叫 MVC、MVP 还是 MVVM，而是职责边界有没有清楚、依赖方向有没有正确、核心业务有没有从 UI 中剥离出来。
