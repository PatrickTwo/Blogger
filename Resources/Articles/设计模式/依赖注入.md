## 一、什么是依赖注入（Dependency Injection，DI）

**依赖注入**是一种设计模式，也是实现控制反转（IoC）的主要手段。它的核心思想是：

> **一个类不应该自己创建它所依赖的对象，而是由外部容器（或调用者）把依赖“注入”给它。**

这样可以让类变得更**松耦合**、更容易**测试**和**维护**。

---

## 二、没有 DI 时的问题

假设有一个 `OrderService`，它需要一个 `IOrderRepository` 来做数据操作：

```csharp
public class OrderService
{
    private readonly IOrderRepository _repository;

    public OrderService()
    {
        // 自己 new 了一个具体实现
        _repository = new SqlOrderRepository();
    }

    public void CreateOrder(Order order)
    {
        _repository.Save(order);
    }
}
```

**问题：**
1. **紧耦合**：`OrderService` 直接依赖 `SqlOrderRepository`，想换成 `MongoOrderRepository` 要改源码。
2. **难以测试**：单元测试时无法替换成模拟对象（Mock）。
3. **职责不清**：类既要处理业务逻辑，又要负责创建依赖。

---

## 三、引入依赖注入

我们把依赖的创建交给外部，类只声明“我需要什么”：

```csharp
public class OrderService
{
    private readonly IOrderRepository _repository;

    // 通过构造函数传入依赖
    public OrderService(IOrderRepository repository)
    {
        _repository = repository;
    }

    public void CreateOrder(Order order)
    {
        _repository.Save(order);
    }
}
```

调用方这样使用：

```csharp
var repository = new SqlOrderRepository(); // 外部创建
var service = new OrderService(repository); // 注入依赖
service.CreateOrder(new Order());
```

这样 `OrderService` 只关心如何使用 `IOrderRepository`，不关心它是怎么来的。

## 四、依赖注入的三种常见方式

### 1. 构造函数注入（最推荐）

```csharp
public class OrderService
{
    private readonly IOrderRepository _repository;

    public OrderService(IOrderRepository repository)
    {
        _repository = repository;
    }
}
```

- 依赖在对象创建时必须提供，保证对象处于**完全可用状态**。
- 适合**必需依赖**。

### 2. 属性注入

```csharp
public class OrderService
{
    public IOrderRepository Repository { get; set; }
}
```

- 通过公共属性设置依赖。
- 适合**可选依赖**，但容易忘记设置，导致运行时错误。

### 3. 方法注入

```csharp
public class OrderService
{
    public void ProcessOrder(Order order, IOrderRepository repository)
    {
        repository.Save(order);
    }
}
```

- 每次调用方法时传入依赖。
- 适合**仅在该方法中使用**的依赖。

## 五、依赖注入容器（IoC Container）

手动 `new` 并传参在简单场景没问题，但在大型项目中，依赖关系可能很复杂，这时可以用**DI 容器**自动管理。

在 ASP.NET Core 中，内置了 DI 容器，使用方式如下：

### 1. 注册服务

```csharp
// Program.cs 或 Startup.cs
services.AddScoped<IOrderRepository, SqlOrderRepository>();
services.AddScoped<OrderService>();
```

- `AddScoped`：每次请求创建一个实例（Web 应用中常用）。
- `AddSingleton`：全局单例。
- `AddTransient`：每次注入都创建新实例。

### 2. 使用服务

在控制器或其他服务中直接声明构造函数参数，容器会自动注入：

```csharp
public class OrdersController : Controller
{
    private readonly OrderService _orderService;

    public OrdersController(OrderService orderService)
    {
        _orderService = orderService;
    }

    public IActionResult Create(Order order)
    {
        _orderService.CreateOrder(order);
        return Ok();
    }
}
```

## 六、依赖注入的好处

1. **解耦**：类之间不直接依赖具体实现，只依赖抽象（接口/基类）。
2. **可测试性**：测试时可以注入 Mock 对象，隔离外部依赖。
3. **可维护性**：更换实现只需修改注册代码，不用改业务类。
4. **符合单一职责原则**：类只关注自己的核心逻辑，不负责创建依赖。

---

## 七、需要避免的坑

- **服务定位器模式滥用**：不要自己在类里 `new ServiceCollection()` 去解析服务，这会破坏 DI 的优势。
- **循环依赖**：A 依赖 B，B 又依赖 A，容器会报错，需要重新设计。
- **过度注入**：不要把所有东西都注入，只有真正需要替换或测试的依赖才通过 DI 注入。
